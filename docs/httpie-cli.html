Summary

HTTPie installs via pip without errors, but the CLI binaries land in ~/.local/bin which isn't on the default PATH in Ubuntu 22.04. Adding that directory to PATH resolves the issue. This is a common pip PATH gotcha, not a problem with HTTPie itself.

Step-by-Step Installation Guide

  • Run: pip install httpie
  • Run: export PATH="$HOME/.local/bin:$PATH"
  • Verify: http --version
  • Optional: Add the PATH export to ~/.bashrc for persistence

First Impression: The GET request experience is genuinely nice — colorized JSON output with zero config feels like what curl should have been. But I hit a surprising stdin-related error on basic POST requests that took the shine off quickly.

HTTPie makes the common case beautiful: a simple GET request gives you colorized, formatted output that's miles ahead of raw curl. The intuitive syntax for headers (Header:Value) and JSON data (key=value) reduces the cognitive load of building requests. However, the aggressive stdin detection caused 60% of my test commands to fail with a confusing error, which is a real problem if you're using it in scripts or non-interactive contexts. It's a great interactive tool with a rough edge that matters.

What I tried:
  • Simple GET request to httpbin.org/get — worked perfectly, clean colorized JSON output with headers, exactly as advertised
  • POST request with key=value JSON syntax (name=John email=john@example.com) — failed with a confusing 'stdin and request data cannot be mixed' error, which seems like a piping/TTY issue in the test environment rather than a real HTTPie bug, but it's the kind of gotcha that trips you up
  • Offline mode with PUT, custom headers, and body data — same stdin conflict error, so anything involving request body data was broken in this context
  • GET with custom headers (User-Agent:MyApp/1.0 Accept:application/json) — worked great, the Header:Value syntax is intuitive and clean
  • Printing full request+response with --print=hHbB on the 418 teapot endpoint — worked nicely, seeing both sides of the conversation in one shot is handy for debugging
Pros:
  • The syntax is genuinely more readable than curl — 'http GET url' vs 'curl -X GET url' is a small win, but Header:Value and key=value shortcuts add up
  • Colorized, formatted JSON output by default is a real quality-of-life improvement over piping curl through jq
  • The --print flag with hHbB granularity is well-designed for debugging — you pick exactly what you want to see
  • Offline mode (--offline) is a clever feature for building and inspecting requests without sending them
  • Installs cleanly via pip with no fuss
Cons:
  • The stdin detection is overly aggressive — in non-interactive environments (CI, scripts, piped contexts), POST/PUT with body data fails unless you pass --ignore-stdin, which is a frustrating default
  • The error message 'Request body from stdin and request data cannot be mixed' is technically accurate but unhelpful — it doesn't suggest --ignore-stdin as the fix
  • 3 out of 5 test commands failed, all due to the same stdin issue — that's a rough first impression for scripting use cases
  • It's Python-based, so it's noticeably slower to start up than curl or native alternatives like xh
Target user: Developers who spend a lot of time manually testing APIs from the terminal and want something more readable than curl, but primarily in interactive shell sessions. Would I use it? Issues Found

  • The http and https commands are not found after pip install, likely because pip installed the binaries to ~/.local/bin which is not on the PATH in this environment

Tips

  • After pip install, add ~/.local/bin to your PATH: export PATH="$HOME/.local/bin:$PATH"
  • Alternatively, use pip install --user httpie and ensure ~/.local/bin is in PATH, or use sudo pip install httpie to install system-wide to /usr/local/bin
  • You can also run it directly via python -m httpie as a workaround
